# The Autonomaton Pattern: Architectural Mandate
You are operating within the repository for "The Coach Autonomaton," a reference implementation of The Grove Foundation's Autonomaton Pattern. 

Your primary directive is to enforce the architectural invariants of this pattern. You are not building a standard agentic AI script; you are building an operating system where logic lives in configuration, and the system executes via an immutable pipeline.

## 1. The Invariant Pipeline
Every interaction MUST traverse the following five-stage pipeline. You are forbidden from writing code that bypasses these stages to execute an action directly:
1. **Telemetry:** Every user utterance or system event is logged as structured data (JSONL).
2. **Recognition:** Telemetry is parsed to identify intent, entities, and domain.
3. **Compilation:** The system queries the local Dock (RAG) and drafts proposed actions.
4. **Approval:** Governance checks are applied. If Yellow/Red zone, explicit one-thumb user approval is required.
5. **Execution:** The action is carried out (via MCP or local file write) ONLY if approval is met.

## 2. Config Over Code (Declarative Sovereignty)
Logic belongs in configuration files, not in executable code. 
* Routing rules live in `routing.config`.
* Governance and boundaries live in `zones.schema`.
* External capabilities live in `mcp.config`.
* Brand/Voice guidelines live in `voice.yaml`.
Before proposing a new feature or behavior, you MUST update the relevant configuration files first.

## 3. Digital Jidoka (No Silent Failures)
You are forbidden from writing "graceful degradation" or silent error handling when facing ambiguity (e.g., entity resolution, unclear intent). You must trigger Jidoka: stop the line and surface a numbered, one-keystroke prompt for the user to resolve the ambiguity.

## 4. MCP as Effectors
External actions (Calendar, Gmail) are handled exclusively via Model Context Protocol (MCP) servers. Every MCP action must be strictly governed by the domain's Zone classification.
